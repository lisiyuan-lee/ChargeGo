## 三、计费费率验证

### 3.1 电价函数逻辑

系统根据当前小时返回对应电价，代码审查确认：

```python
def electricity_rate(moment):
    h = moment.hour
    if 10 <= h < 15 or 18 <= h < 21: return 1.0   # 峰时
    if 7 <= h < 10 or 15 <= h < 18 or 21 <= h < 23: return 0.7  # 平时
    return 0.4  # 谷时
```

### 3.2 计费公式

```
总费用 = 充电电量 × 电价(当前时段) + 充电电量 × 服务费率(0.8)
```

### 3.3 实测验证

使用测试用户提交快充请求，充电一段时间后结束并生成账单：

| 项目 | 值 |
|------|------|
| 充电电量 | 5.92 kWh |
| 电费 | ¥5.92（= 5.92 × 1.0，当前为峰时） |
| 服务费 | ¥4.74（= 5.92 × 0.8） |
| 总费用 | ¥10.66 |
| 有效总费率 | **¥1.8000/kWh**（电价1.0+服务费0.8=1.8） |

各时段费率验证：

| 时段 | 电价 | 服务费 | 总费率 | 结果 |
|:----:|:----:|:-----:|:------:|:----:|
| 谷时 | 0.4 | 0.8 | 1.2 | ✅ 代码逻辑正确 |
| 平时 | 0.7 | 0.8 | 1.5 | ✅ 代码逻辑正确 |
| 峰时 | 1.0 | 0.8 | 1.8 | ✅ 实测 ¥1.8000 |

**结论：计费公式完全正确 ✅**

## 五、多车辆并发测试

### 5.1 场景一：6辆快充 + 4辆慢充同时提交

注册10个测试用户（multi01~multi10），同时提交充电请求：

| 充电桩 | 充电中 | 排队 | 总负载/上限 | 结果 |
|:------|:-----:|:----:|:----------:|:----:|
| FastCharger-1 | 1辆 | 2辆 | 3/3 | ✅ 已满 |
| FastCharger-2 | 1辆 | 2辆 | 3/3 | ✅ 已满 |
| SlowCharger-1 | 1辆 | 1辆 | 2/3 | ✅ |
| SlowCharger-2 | 1辆 | 0辆 | 1/3 | ✅ |
| SlowCharger-3 | 1辆 | 0辆 | 1/3 | ✅ |

验证结果：
- **负载均衡**：快充车辆交替分配到两个桩（F5→桩1, F6→桩2, F7→桩1,...），慢充车辆均匀分配到三个桩
- **队列深度控制**：每个桩均未超过3辆上限

### 5.2 场景二：队列满→等候区溢出

在快充队列已满（6辆车占满2个桩）的情况下，再提交3辆快充请求：

| 车辆 | 请求编号 | 分配状态 | 结果 |
|:----|:--------:|:--------:|:----:|
| multi11 | F11 | waiting（等候区） | ✅ |
| multi12 | F12 | waiting（等候区） | ✅ |
| multi13 | F13 | waiting（等候区） | ✅ |

等候区 API 返回正确显示3辆等待中。

### 5.3 场景三：故障 + 多车辆重分配

关闭 FastCharger-1（strategy=sequence），受影响3辆车：

| 车辆 | 原状态 | 处理后状态 | 结果 |
|:----|:------:|:----------:|:----:|
| multi01(F5) | 充电中 | 已结算(cancelled) | ✅ |
| multi03(F7) | 排队中 | 故障等待(fault_waiting) | ✅ |
| multi05(F9) | 排队中 | 故障等待(fault_waiting) | ✅ |

恢复 FastCharger-1 后，调度器自动处理故障等待车辆。
